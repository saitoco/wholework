# auto-session Narrative Prompts

Prompt templates for LLM-assisted narrative draft generation in `/audit auto-session --full`.

Each section below corresponds to one narrative section in the retrospective report.
The LLM reads the data-layer report (already generated by Step 1) plus these prompts to produce drafts.

---

## Prompt Templates

### What worked

```
Based on the data-layer report below, extract 3-5 elements that functioned as designed.
Focus on: recovery events that resolved cleanly, phases that completed within expected time,
metrics that match design intent, and patterns that held under stress.
For each element, state the observation concisely and, where data supports it, include a
specific number (e.g., "2 of 35 wrapper phases required Tier 3 recovery (5.7%)").
Do not speculate beyond what the data shows. Output as a numbered list.
```

### Limits and gaps

```
Based on the data-layer report below, extract 3-5 structural observations where the system
fell short, accumulated debt, or revealed a design gap.
Focus on: terminal states that required manual intervention, metrics that deviated from
design intent, patterns that appeared more than once, and gaps between what was automated
and what was not.
For each item, describe the observation and its scope (isolated vs. systemic).
Do not propose fixes here — that belongs in "Improvement candidates surfaced".
Output as a numbered list.
```

### Improvement candidates surfaced

```
Based on the "Limits and gaps" items you identified, generate one improvement candidate per
structural gap. Each candidate should describe: (1) the problem, (2) a concrete fix direction,
and (3) a classification:
  - "既存 #XXX に統合提案" — if a related open issue already covers this (search with gh issue list --search)
  - "Issue 起票候補" — if this is a new improvement worth filing, include a one-paragraph issue body skeleton
  - "凍結推奨（trigger: XXX）" — if the fix is premature pending another event or decision
Output each candidate as a numbered list item with its classification clearly marked.
Note: do not file issues automatically — this section is for human review and decision.
```

### Conclusion

```
Based on the full session data and the narrative sections above (What worked, Limits and gaps,
Improvement candidates surfaced), write a 2-3 paragraph conclusion that:
1. Summarizes the session's overall result (throughput, recovery health, key accomplishments)
2. Names the most important structural finding (the gap or risk that most deserves attention)
3. States what the session demonstrates about Wholework's current state

Write in present-tense, factual prose. Avoid hedging language. Ground every claim in the data.
```

## Few-shot Examples

The following excerpts from past retrospective reports illustrate the quality and style expected.
Reference these as examples when generating drafts.

### What worked (from 2026-06-13 report)

1. **watchdog + 3-tier recovery**: 2 kills in 35+ wrapper phase runs (5.7%). One converted a false positive into a structural fix (#569); the other, post-#569, correctly detected a true stall. Beyond "stable" — a self-diagnosing system.
2. **verify FAIL → reopen → fix cycle**: #557 resolved within the iteration cap (2/3). The CI-red verify correctly caught the code phase's inaccurate self-report of green.
3. **retro → Issue → /auto self-repair**: #557 incident → #569 fix closed in under 10 hours. Outflow (5 issues filed) = inflow (5 consumed). The design goal demonstrated in practice.
4. **Triage's unintended secondary value**: 3 issues had defective verify commands (false PASS / false FAIL) repaired at triage time — value beyond its stated responsibility (metadata assignment).
5. **Dynamic Size reassessment (Step 3a)**: #560's M→S re-judge worked, route switched to patch, completed in 45 min.

### Limits and gaps (from 2026-06-13 report)

1. **Observation-type post-merge AC accumulation**: 7 of 14 issues remain at phase/verify (CLOSED but not fully done). "Future-event observation" and "opportunistic consumption" are mixed and accumulate.
2. **Manual dependency in fix cycle**: #557's parent session manually fixed 5 test-mock files — beyond `/code`'s self-repair scope. Detected only after a 45-minute empty wrapper run; drags down autonomous completion rate.
3. **Implicit triage audit of verify commands**: All 3 repairs were incidental discoveries by the triage executor. Not documented as a skill responsibility, no systematic guarantee.
4. **Excessive headroom in uniform 2700s watchdog**: Longest observed silent windows: review 660s / code 480-540s / spec 480-540s — 4-5x headroom. True stalls detected slowly.

### Improvement candidates surfaced (from 2026-06-14 report)

1. **Batch wrapper verify orchestration** — "Issue 起票候補": Either invoke `/verify` per child issue from the parent session after each `run-auto-sub.sh` returns, or document that batch mode always terminates at `phase/verify` and requires follow-up `/verify N`. The current behavior is correct but silently produces a backlog.
2. **In-flight route demotion on Step 3a re-judge** — "凍結推奨（trigger: Step 3a complexity budget review）": When spec re-judges Size from M to XS, the remaining phases could be re-planned to patch route. Trade-off is added complexity in the phase orchestrator.
3. **Tier 3 sub-agent invocation logging** — "既存 #316 に統合提案": The trace was captured in the wrapper log only. A summary line in `docs/reports/orchestration-recoveries.md` keyed on `source: recovery-sub-agent` would close the audit-trail loop.

### Conclusion (from 2026-06-14 report)

The `--batch` List mode run of 6 issues completed cleanly under an Opus 4.7 parent over 4h 43m of continuous execution. The two notable production firsts are (1) a Tier 3 recovery sub-agent invocation that resolved without parent intervention, and (2) coexistence with a concurrent single-session `/auto` on the same repository without merge conflict. Throughput at ~1.27 issues/hr is consistent with the M-heavy issue mix.

The unresolved structural observation is the universal terminal `phase/verify` state for batch-completed issues — a wrapper-design gap that the AC-side fix (#583, event-driven observation type) does not address. Either batch mode should orchestrate verify between issues, or the design should be acknowledged as "batch terminates at verify, user runs `/verify N` per follow-up."
