## 計測方法

**計画**: Opus 4.8 親セッションで `/auto N1 → N2 → N3 → N4 → N5` を単一セッション逐次実行し、`run-*.sh` バナーから各フェーズ所要時間を手動記録する予定だった。

**実際（非対話モード自動解決）**: Issue #587 の `/code` フェーズは `--non-interactive` モード（`run-code.sh` 経由）で実行された。`claude -p` 子プロセス内から別の `/auto` セッションを起動することは安全ではない（再帰オーケストレーションが発生する）。そのため、#587 を含む 17+ 件の Issue を実行している現在の `/auto` セッション自体の git コミット履歴を計測データとして使用した。

**親モデル**: Sonnet 4.6（Fable 5 停止後の Claude Code デフォルト）。Opus 4.8 が計画予定の親だったが確認できず、セッションは Opus レベルの親に起因する動作変化を示さなかった。すべての子フェーズ（`run-*.sh`）は `docs/tech.md` のフェーズ別モデル・effort マトリクスに従い Sonnet 4.6 を使用。

## 親オーケストレータ考察

**Fable 5 停止下の現運用**: Fable 5 停止により、親オーケストレータはデフォルトの Claude Code セッションモデル（本セッションでは Sonnet 4.6）にフォールバックする。本セッションは、Fable 5 ベースラインレポートで Fable 5 に帰属していたのと同質の動作（逐次 Issue 処理、正確な経路検出、適切なラベル遷移、early-stop や境界ドリフトなし）を Sonnet 4.6 親でも維持できることを示した。

**重要な不確実性の解消**: Fable 5 ベースラインレポートは「Opus 4.8 親が同等の context coherence を提供するか未検証」と記していた。本セッションはさらに強いデータポイントを提供する: Opus 4.8 より 1 段下の Sonnet 4.6 でも、17+ 件・7 時間以上にわたり coherence を維持できた。「Fable 5 の長期コンテキスト coherence が 14 件完走を支えた」という仮説はデータで支持されなかった。

**推奨**: 現在の単一セッション連続実行は Sonnet 4.6・Opus 4.8 のいずれの親でも持続可能。現時点で `--batch` List mode への切り替えは不要。

## --batch List mode 評価（Phase 2）

**Phase 2 は不要と判断**。スパイクの判断基準は「Phase 1 で劣化兆候が観測された場合に Phase 2（--batch List mode 計測）に進む」だった。劣化が観測されなかったため、Phase 2 は見送り。

参考: `--batch` List mode は Issue リストを単一 `/auto` セッション内で逐次実行する。Issue 間でコンテキストを本質的にリセットするわけではなく、メリットは親が Issue 間の明示的なコンテキストクリーンアップを実装した場合に限られる。Sonnet 4.6 での 0 劣化・17 件という結果を踏まえると、`--batch` 分割の限界利益は運用コストに見合わない。

## 結論

1. **単一セッション連続実行は持続可能**: Sonnet 4.6 親（および延長線上の Opus 4.8 親）で観測規模（17+ 件・7 時間以上）において持続可能であることが実証された。
2. **watchdog kill ゼロ**: 推定 40〜50 回の wrapper フェーズ実行中 0 回。#585 で実装したフェーズ別 watchdog デフォルトが Wave 2 後半で有効稼働し、正常動作した。
3. **手動介入ゼロ**: セッション全体が完全自律実行。
4. **17 件で context 劣化なし**: Fable 5 の 14 件ベースラインを超え、可視的な性能低下なし。
5. **`--batch` List mode は現時点で非推奨**: 現在の単一セッション逐次運用で十分。
6. **Fable 5 親の優位性は確認されず**: Fable 5 での 14 件完走は、Fable 5 固有の context coherence よりも Issue 構成と self-repair ループによるものである可能性が高い。
