# Sonnet 5 Effort Recalibration: `run-spec.sh`

**Report date**: 2026-07-05
**Issue**: #922 (C3, `docs/reports/claude-sonnet-5-impact-strategy.md` §8)
**Scope**: Re-evaluate whether `run-spec.sh`'s Sonnet-path default effort can drop from `max` to `xhigh` under Sonnet 5's widened effort curve, including a comparison against the existing Opus fallback (`--opus`, L-size only).
**Depends on**: `docs/reports/claude-sonnet-5-impact-strategy.md` §3.3/§4.2/§4.6 (candidate framing); Issue #217 (2026-xx-xx, Opus path `max`→`xhigh` swap — Sonnet path explicitly out of scope); Issue #229 (`docs/reports/sonnet-effort-recalibration.md`, Sonnet 4.6-era evaluation of `run-code.sh`/`run-review.sh`/`run-merge.sh`/`run-verify.sh`/`run-issue.sh` — `run-spec.sh` explicitly excluded); `docs/reports/sonnet-5-effort-recalibration-code-review.md` (#921, C2, sibling report and methodology precedent).

## Background

C1 (#914, default parent swap to Sonnet 5) landed, making phase-specific effort re-evaluation actionable. The impact strategy report §3.3 argues Sonnet 5 widens the cost/performance curve: `medium` becomes more competitive with `high` for tasks that don't need deep reasoning, and `xhigh` approaches Opus 4.8-class performance on a meaningful share of tasks. §4.2 frames `run-spec.sh` (currently `max` for the Sonnet default path) as a candidate worth re-checking, but explicitly flags it as "quality-sensitive... not a default flip" — spec errors propagate to every downstream phase (code, review, merge), which §4.6 calls out as the costliest place in the whole C-series for a regression to land.

Unlike `run-code.sh`/`run-review.sh` (#921, C2), `run-spec.sh`'s Sonnet-path default effort (`max`) has never been formally evaluated at any point in this repository's history — confirmed below (Evaluation Method #1). This report is the first such evaluation.

No quantitative A/B benchmarking harness exists for isolating effort-level impact on spec-quality for a single task (duplicating a Spec run across effort levels for the same Issue is not practical in production). Per the Issue's Auto-Resolved Ambiguity Points, this report follows the `#877`/`#921` precedent: combine (a) qualitative structural analysis of the `/spec` workload, (b) a check of whether the Anthropic guidance actually documented in this repository (`docs/tech.md`'s "Opus 4.8 effort calibration" note) generalizes to Sonnet 5, and (c) a production-sample check of recent Spec-driven Code Retrospectives as a proxy signal for whether the current `max` tier shows headroom.

## Evaluation Method

1. **Re-derive #217/#229's scope decisions.** Confirm whether either precedent Issue evaluated the Sonnet default path's `max` effort, and if not, why not.
2. **Structural analysis of `skills/spec/SKILL.md`.** Determine whether `/spec` is a single continuous reasoning chain (like `run-code.sh`) or delegates deep analysis to sub-agents (like `run-review.sh`'s Opus sub-agents) — this determines which of #921's two argument patterns applies.
3. **Check the sourcing of the "`max` has diminishing returns" claim.** `docs/tech.md`'s "Opus 4.8 effort calibration" note is the only documented diminishing-returns guidance in this repository. Verify whether it is scoped to Opus 4.8 specifically or generalizes to Sonnet 5's `max` tier.
4. **Production-sample check.** Survey `docs/spec/*.md` Code Retrospective "Design Gaps/Ambiguities" entries, filtered to Issues coded after Sonnet 5 became the default parent (#914, i.e. Issue numbers ≥ 915), as a proxy signal for whether the current `max`-effort baseline shows over-provisioning (frequent, substantive design-level gaps would suggest current effort is already insufficient — the opposite direction from a downgrade case; an absence of any gaps at all across many samples would suggest possible headroom).
5. **Opus fallback comparison.** Compare the existing `--opus` (L-size) path's `xhigh` default (set in #217, per Opus 4.7's own calibration guidance) against the Sonnet default path's `max`, and assess whether the Opus precedent transfers as evidence for a Sonnet downgrade.

## Analysis 1 — Precedent re-derivation: has Sonnet-path `max` ever been evaluated?

**#217** (Opus 4.7 introduces `xhigh`): switched only the `--opus` path's default from `max` to `xhigh`. The Issue's own Purpose statement is explicit: "非 Opus (Sonnet) パスは本 Issue のスコープ外 (現状維持)" (the non-Opus/Sonnet path is out of scope for this Issue — left unchanged). No Sonnet-path evaluation occurred.

**#229** (`docs/reports/sonnet-effort-recalibration.md`, Sonnet 4.6 era): evaluated the Sonnet effort tier (`high`/`medium`/`low`) for `run-code.sh`, `run-review.sh`, `run-merge.sh`, `run-verify.sh`, `run-issue.sh`. The Issue body's own Background explicitly excludes `run-spec.sh`: "`xhigh` effort は Opus 専用のため... run-spec.sh --opus 以外に直接は適用されない" (at the time, `xhigh` was Opus-only, so the Opus-4.7-driven recalibration trigger did not apply to any of the five Sonnet-tier scripts under evaluation, and `run-spec.sh`'s Sonnet path — which uses the separate `max` tier, not the `high`/`medium`/`low` ladder the other five scripts use — was not part of that ladder-recalibration exercise either).

**Verdict of this check**: confirmed — `run-spec.sh`'s Sonnet-path `max` default has never been evaluated across two prior recalibration passes (Opus-triggered in #217, Sonnet-ladder-triggered in #229). This Issue (#922) is the first formal evaluation.

Separately, `scripts/run-spec.sh`'s actual implementation was re-confirmed to match `docs/tech.md`'s matrix row exactly: `EFFORT="max"` default (line 12), `--opus` sets `MODEL="opus"` and `EFFORT="xhigh"`, `--fable` sets `MODEL="claude-fable-5"` and `EFFORT="high"`, and `--max` force-overrides `EFFORT="max"` regardless of model. This matches the matrix row "Sonnet: max; Opus: xhigh (default), max (explicit `--max`); Fable 5: high (default), max (explicit `--max`)" — no SSoT drift found (Issue AC "A").

## Analysis 2 — Structural comparison with `run-code.sh`/`run-review.sh` (#921)

`skills/spec/SKILL.md` runs as a single agent with **no sub-agent fan-out** for its core investigation and design work (confirmed via grep — no `Agent(...)`/`Task(subagent_type=...)` calls in the skill body; the L/XL parallel investigation sub-agents `issue-scope`/`issue-risk`/`issue-precedent` listed in `docs/tech.md`'s matrix belong to `/issue` Step 11a, not `/spec`). It executes a 19-step reasoning chain in one continuous session: mode detection → Issue fetch → worktree entry → label transition → blocked-by detection → steering-doc reference → codebase investigation (with sub-checks for credential-policy alignment, issue/implementation conflict detection, tool-detection-pattern consistency, dependency-version verification, adapter-pattern survey) → ambiguity resolution → uncertainty identification → UI design phase → Spec creation → title drift check → commit → retrospective → worktree exit → comment → label transition.

This is the **same structural pattern** #921 used to justify maintaining `high` for `run-code.sh`: a single, unbroken multi-step reasoning chain with no sub-agent fan-out to independently re-check a shallow-reasoning error introduced early in the chain. The consequence is also structurally worse here than for `run-code.sh`: a shallow-reasoning error in `run-code.sh` surfaces as rework within the same PR (caught by review/tests before merge); a shallow-reasoning error in `run-spec.sh` ships as the Implementation Steps that `run-code.sh` then executes largely at face value — the error propagates through code, review, and merge before it can be caught, exactly as the Issue's own Background and the impact report's §4.6 both state ("spec errors propagate to all downstream phases... the phases where a regression is costliest").

## Analysis 3 — Does the "`max` has diminishing returns" claim generalize to Sonnet 5?

`docs/tech.md`'s only documented diminishing-returns guidance reads: "Opus 4.8 enforces strict effort calibration — `low` and `medium` aggressively scope to literal task requirements. `max` carries a diminishing returns risk (overthinking) with **Opus 4.8**... `xhigh` is the **Opus 4.8** recommended default." This note is scoped explicitly to Opus 4.8 model behavior, not to Sonnet 5.

The impact strategy report's §3.3 Sonnet-5-specific claim is narrower: `medium` effort "delivers meaningfully better cost efficiency for tasks that do not need deep reasoning," and `xhigh` "approaches Opus 4.8-class performance on a subset of tasks." Neither of these Sonnet-5-specific claims asserts that Sonnet 5's own `max` tier suffers a diminishing-returns/overthinking penalty relative to Sonnet 5's own `xhigh` — that framing exists only for Opus 4.8 in this repository's documentation. The Issue's premise ("`xhigh` が Opus 4.8 に迫る framing suggests `xhigh` may be sufficient") is the impact report author's own speculative extrapolation, not a claim sourced from Anthropic's official Sonnet 5 guidance recorded anywhere in `docs/tech.md` or `docs/reports/claude-sonnet-5-impact-strategy.md`.

**Reading**: the evidentiary basis for downgrading is weaker here than it was for `run-code.sh`/`run-review.sh` in #921. In #921, the impact report itself scoped a `medium` candidacy for `run-code.sh` (XS/S patch-route only) and characterized `run-review.sh`'s orchestration as "mechanical" — both claims turned out to be inaccurate on closer inspection, but they were at least *specific, falsifiable* claims about those two scripts. For `run-spec.sh`, the impact report offers no equivalently specific claim beyond a general "xhigh approaches Opus" framing that is not shown to apply to Sonnet 5's own effort-tier internal ordering.

Absence of both a quality-degradation signal and a cost/time-complaint signal leaves this check net-neutral: it does not support a downgrade, and it does not contradict one either — it simply confirms `max` is not currently a documented pain point.

## Analysis 5 — Opus fallback (`--opus`, L-size) comparison

The `--opus` path has used `xhigh` as its default since #217 (Opus 4.7's own official recommendation: "`xhigh` is the Opus 4.8 recommended default for most coding and agentic use cases," `docs/tech.md`). This is sometimes read as informal evidence that "`xhigh` is good enough for the highest-risk spec workload (L-size)," which could be read as transferable to the Sonnet default path.

This transfer does not hold under scrutiny: the Opus `xhigh` default reflects Opus 4.8's *own* documented effort-calibration guidance (where `max` is specifically flagged as an overthinking risk *for Opus*) — it says nothing about where Sonnet 5's effort curve places `max` versus `xhigh` for the same workload. Effort-tier labels are not calibrated identically across model families; `docs/tech.md`'s "Opus 4.8 effort calibration" and "Sonnet 5" notes are two separate paragraphs precisely because the two model generations' effort curves are documented as distinct. Additionally, the `--opus` path is invoked only for L-size specs (`/auto` passes `--opus` for L only, per the matrix row's own rationale column) — a narrower, higher-scrutiny slice of work than the Sonnet default path, which spans all Issue sizes (XS through L, with `--opus` reserved for L). Using the L-size-only Opus calibration to justify a change to the all-sizes Sonnet default path would repeat the same size-scoping mismatch #921 found for `run-code.sh` (where the impact report's `medium` candidacy was scoped to XS/S but the flag being evaluated for change is global).

**Conclusion of this comparison**: the Opus fallback's `xhigh` default is not evidence that the Sonnet default path's `max` is over-provisioned. The two data points are not comparable across model generation or Issue-size scope.
