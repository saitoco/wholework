# Issue #694: audit: batch/XL 完了時の auto-session retrospective (L3) を自動化 + docs/sessions/ レイアウト導入

## Overview

`/auto --batch` または XL 完了時に L3 session retrospective を自動 trigger する。notable な cross-cutting observation がある場合のみ `docs/sessions/{SID}-{DATE}/` を生成し、`modules/retro-proposals.md` 経由で改善 Issue を起票する。単一 `/auto N` (M/L/patch) では L3 は不要 (L4 per-Issue retrospective と重複するため skip)。

- `skills/auto/SKILL.md`: Step 5 の "Daily rollup" の直後に L3 auto-retrospective block を追加 — bash 3.2+ compatible
- `docs/structure.md`: Directory Layout に `docs/sessions/` エントリを追加
- `docs/ja/structure.md`: 英語 `docs/structure.md` の変更を日本語 mirror に反映

## Notes

### Auto-Resolved Ambiguity Points (non-interactive mode)

Issue 本文に `## Auto-Resolved Ambiguity Points` セクションとして記録済み。Spec での再掲:

1. **concurrent commits >= X件 の X**: X = 3 を採用。`AUTO_MAX_CONCURRENT` デフォルト 5 に対し、3 件以上の同時コミットは重複リスクを示す合理的な閾値。
2. **SID の形式**: `{PID}-{EPOCH}` 形式 (`AUTO_SESSION_ID`) — Step 1 の `SESSION_ID="$$-$(date +%s)"` と一致。
3. **retro-proposals.md シグネチャ変更スコープ**: 本 Issue 外 (#710 で対応)。現 interface をそのまま使い、設計コメントを SKILL.md に残す。

### retro-proposals.md Interface Bridge

`retro-proposals.md` は `Glob with $SPEC_PATH/issue-$NUMBER-*.md` で Spec ファイルを特定する。L3 session retrospective は通常の Spec ファイル形式ではないため、bridge file を `$SESSION_DIR/issue-{BRIDGE_NUMBER}-l3session.md` として作成し、SPEC_PATH を `$SESSION_DIR` に一時設定してから呼び出す。BRIDGE_NUMBER は XL の場合は親 Issue 番号、batch の場合は `"batch-{AUTO_SESSION_ID}"` (文字列) を使用する。重複チェック (`gh issue list`) はルーティング後の Issue タイトル比較で機能するため、BRIDGE_NUMBER が数値でなくても問題ない。

### Route Detection for ROUTE Variable

Step 5 の L3 guard は Step 2/3 で確定した `ROUTE` 変数を参照。batch mode は Step 1 で `--batch` flag 検出後に Steps 2-6 をスキップするため、batch 完了報告後に L3 block を呼ぶ際は `ROUTE="batch"` を明示的にセットしておく必要がある (Implementation Step 1 内に記載)。

### Commit Count Detection

"concurrent commits >= 3件" の判定は、events ログから `event_type == "commit"` (または類似) のエントリ数をカウントする。events.jsonl フォーマットが commit イベントを emit しない場合は、`git log --oneline --since="$session_start"` の行数を代替として使用する。Implementation 時に events log schema を確認して適切な検出方法を選択すること。

### docs/sessions/ Migration

既存の `docs/reports/auto-session-*.md` は旧 `/audit auto-session` の手動レポート。migration は別 Issue で扱う (Issue 本文に明記)。

### #704 / #710 統合設計コメント

- #704 (autonomy tier): 実装時は route guard のみ (`batch || XL`) で L3 を trigger。autonomy 判定は混ぜない。#704 着地後に 1 行 guard を追加するだけで済む形にしておく。
- #710 (blocked-by workflow): SKILL.md の retro-proposals 呼び出し箇所に `<!-- TODO (#710) -->` コメントを残す。正式 relationship 設定への切り替えは #710 で行う。

## Code Retrospective

### Deviations from Design

- Spec では route guard を `ROUTE == "XL"` と記述していたが、SKILL.md の既存 Step 3a route table では XL は `sub_issue` を使用するため、guard 条件を `ROUTE == "sub_issue"` に変更した。"XL" は Spec の意図する名称で `sub_issue` が実際の変数値であり整合性を優先した。
- Spec では deprecated term の修正スコープは記載されていなかったが、既存 Spec Notes に含まれていたため /code フェーズ内で修正した。

### Design Gaps/Ambiguities

- Spec の "Route Detection for ROUTE Variable" Notes では `ROUTE="batch"` を Batch Completion Report で設定する必要があると述べているが、L3 block の配置場所として Step 5 のみ指示されていた。batch mode は Steps 2-6 をスキップするため Batch Completion Report に L3 block を複製追加 (Event scan + Daily rollup + ROUTE="batch" + L3 参照) した。Spec の実装意図と一致している。
- Spec の Implementation Step 1 コードサンプルに `git push origin main` が含まれていたが、セッション内の commit/push pattern は retrospective commit 時に SKILL.md 外で行われるため、push はコードフェンス内のサンプルとして残しコメントなし。

### Rework

- Spec ファイルの deprecated term 修正後に `git status` で未コミット変更として残り、PR 作成後に追加コミットを要した。spec フェーズが作成した内容の CI scan 対象への該当をコード実装前に確認すべきだった。

### Spec vs. Implementation Divergence Patterns

- Code Retrospective セクションに forbidden expression (旧称) が引用テキストとして残存し、CI Forbidden Expressions check が FAILURE になった。Retrospective で deprecated term を name で引用する場合は `旧称:` 記法や言い換えを使うことを Spec の作成規約に明記する価値がある。

### Recurring Issues

- 同一 PR 内で /code フェーズが「deprecated term 修正をコード後に発見し追加コミット」、/review フェーズが「Code Retrospective 内の deprecated term 引用で CI FAILURE」と、同じカテゴリの問題が 2 回発生した。Retrospective Guard (skill-dev-recheck.md) を /code フェーズにも適用するか、Spec 生成時に forbidden expression scan を実行すると根本対処になる。

### Acceptance Criteria Verification Difficulty

- rubric AC (AC3) はセマンティック判定のため /review の safe mode でも通過できたが、implementation が XL route の variable name を `sub_issue` (Spec では "XL") に変えていることを rubric grader が検出できない可能性がある。rubric text に「ROUTE 変数値 `sub_issue` が XL route の guard として使われている」と具体的変数名を含めると精度が上がる。
