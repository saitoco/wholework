# Issue #58: docs: 日本語ドキュメント自動生成

## Overview

英語ドキュメント（README.md、Steering Documents、Project Documents）を原本として、`/doc translate` サブコマンドで LLM により日本語翻訳を全量再生成し、`docs/ja/` 配下（README のみ `README.ja.md` としてルート直下）に配置、さらにコミット・プッシュまで自動化する。

翻訳対象の発見は既存の Document Traversal（`type: steering|project` の frontmatter 検索）を再利用し、README.md をルート固定ファイルとして追加取り込みする。`docs/ja/` 配下の翻訳ファイルはフロントマターを持たない純粋な Markdown ミラーとし、`/doc sync --deep` の scan からは明示的に除外する。

## Alternatives Considered

- **翻訳ファイルにフロントマターを保持**: 元ファイルの `type: steering|project` を翻訳先にもコピーする案。→ 却下。Document Traversal common procedure が `type:` を Grep で検出するため、SSoT が重複し `status display`・`sync` の動作を壊す
- **サフィックス方式（`docs/product.ja.md`）**: `docs/` 直下に `.ja.md` として配置する案。→ 却下。Document Traversal が `docs/*.md` パターンで拾う設計のため、翻訳ファイルも候補に混入する。ディレクトリ分離（`docs/ja/`）の方が除外ロジックが明確
- **GitHub Actions による自動トリガー**: docs 変更時に自動で翻訳を生成する案。→ 却下。Issue で「任意のタイミングで生成を実行」と明示されており、手動トリガーが要件
- **差分翻訳**: 英語ドキュメントの変更箇所のみ再翻訳する案。→ 却下。LLM 翻訳で差分管理は複雑すぎ、初回実装の範囲外

## Uncertainty

- **LLM 翻訳の品質一貫性**: 全量再生成方式のため、再実行ごとに用語選定や文体が微変動する可能性がある
  - **検証方法**: 初回実行後に生成物を人間レビューで確認し、用語集を追加すべきか判断
  - **影響範囲**: Implementation Step 5（translate セクションの指示）— 用語辞典機能は本 Issue のスコープ外だが、将来拡張として認識しておく
- **フロントマター除去時のメタ情報欠落**: 翻訳ファイルにフロントマターがないため、Document Traversal の型判別対象外となる
  - **検証方法**: `/doc sync --deep` の scan 除外が正しく効くかを post-merge で確認
  - **影響範囲**: Implementation Step 4 の除外条件追記が効くこと。Step 5 の translate セクションでフロントマター除去を明記する

### LLM Translation Instructions

translate セクション内の Step 3（翻訳指示）では、以下の保持ルールを明記すること:

- Markdown 構造（見出し階層・リスト・テーブル・コードブロック）を保持
- コードフェンス内および inline code（backtick）の内容は翻訳せず原文保持
- リンク URL は原文保持、リンクテキストのみ翻訳
- HTML コメント（`<!-- ... -->`）は原文保持（受入検証用ヒント等）
- 源ファイルのフロントマター（`---`〜`---`）は翻訳出力から除去
- 固有名詞・プロダクト名（Wholework、Claude Code 等）は原文保持

### CLAUDE.md Language Conventions 適合性

`CLAUDE.md > Language Conventions` では "Source code / Documentation: English" を規定しているが、本 Issue は翻訳生成物を docs/ja/ 配下の派生物として扱うため、原本の英語ドキュメントは影響を受けない。Non-Goals の例外追記により整合性を確保する。

### settings.json 更新は不要

`modules/skill-dev-checks.md > settings.json Addition Check` は新規 skill 追加時のルールであり、既存 `/doc` skill への追加変更は対象外。また、`.claude/settings.json` は wholework 固有スクリプトのパス allowlist のみを管理しており、git/gh 系の Bash パターンは登録されていない（他の skill も SKILL.md の `allowed-tools` で制御している）。
