# Issue #58: docs: 日本語ドキュメント自動生成

## Overview

英語ドキュメント（README.md、Steering Documents、Project Documents）を原本として、`/doc translate` サブコマンドで LLM により日本語翻訳を全量再生成し、`docs/ja/` 配下（README のみ `README.ja.md` としてルート直下）に配置、さらにコミット・プッシュまで自動化する。

翻訳対象の発見は既存の Document Traversal（`type: steering|project` の frontmatter 検索）を再利用し、README.md をルート固定ファイルとして追加取り込みする。`docs/ja/` 配下の翻訳ファイルはフロントマターを持たない純粋な Markdown ミラーとし、`/doc sync --deep` の scan からは明示的に除外する。

## Alternatives Considered

- **翻訳ファイルにフロントマターを保持**: 元ファイルの `type: steering|project` を翻訳先にもコピーする案。→ 却下。Document Traversal common procedure が `type:` を Grep で検出するため、SSoT が重複し `status display`・`sync` の動作を壊す

- **GitHub Actions による自動トリガー**: docs 変更時に自動で翻訳を生成する案。→ 却下。Issue で「任意のタイミングで生成を実行」と明示されており、手動トリガーが要件
- **差分翻訳**: 英語ドキュメントの変更箇所のみ再翻訳する案。→ 却下。LLM 翻訳で差分管理は複雑すぎ、初回実装の範囲外

### LLM Translation Instructions

translate セクション内の Step 3（翻訳指示）では、以下の保持ルールを明記すること:

- Markdown 構造（見出し階層・リスト・テーブル・コードブロック）を保持
- コードフェンス内および inline code（backtick）の内容は翻訳せず原文保持
- 外部 URL（`http://`, `https://`）は原文保持、リンクテキストのみ翻訳
- HTML コメント（`<!-- ... -->`）は原文保持（受入検証用ヒント等）
- 源ファイルのフロントマター（`---`〜`---`）は翻訳出力から除去
- 固有名詞・プロダクト名（Wholework、Claude Code 等）は原文保持

### Relative Link Rewriting Rules

翻訳出力の配置先が変わるため、相対リンクの書き換えが必要。Step 3 の翻訳処理内で以下のルールを適用する。

**ルール 1: 階層調整（`docs/{name}.md` → `docs/ja/{name}.md` のケース）**

出力ディレクトリが 1 段深くなるため、`../` で始まる相対パスは `../../` に書き換える。

| 元（`docs/workflow.md` 内） | 翻訳先（`docs/ja/workflow.md`）への書き換え |
|---|---|
| `../skills/code/SKILL.md` | `../../skills/code/SKILL.md` |
| `../modules/size-workflow-table.md` | `../../modules/size-workflow-table.md` |
| `../README.md` | `../../README.md`（翻訳版未存在時）または `../../README.ja.md`（翻訳版存在時、ルール 2 参照） |
| `../CLAUDE.md` | `../../CLAUDE.md` |

同一ディレクトリ内リンク（例: `docs/product.md` の `[workflow.md](workflow.md)`）は相対パスが変わらず維持される（`docs/ja/product.md` から `workflow.md` は `docs/ja/workflow.md` を指し、翻訳先に正しく追従）。

**ルール 2: 翻訳版の存在チェック（リンク先差し替え）**

リンク先のファイルが翻訳対象に含まれている場合、翻訳版パスに差し替える。Step 1 で収集した翻訳対象リストを参照して判断する。

| 元（`README.md` 内） | 翻訳先（`README.ja.md`）への書き換え |
|---|---|
| `docs/structure.md` | `docs/ja/structure.md`（翻訳対象のため差し替え） |
| `LICENSE` | `LICENSE`（翻訳対象外のため保持） |

**ルール 3: 書き換え対象外**

- 外部 URL（`http://`, `https://` 始まり）
- アンカーのみのリンク（`#section-name`）
- 翻訳対象外ファイルへのリンク（Step 1 の対象リストに含まれないファイル）

**ルール 4: 適用順序**

1. ルール 1（階層調整）を先に適用
2. ルール 2（翻訳版差し替え）を後に適用して、書き換えた `../../` パスのうち翻訳版が存在するものを `../../docs/ja/{name}.md` に差し替える

**実装時の留意点**: 書き換えロジックは SKILL.md の translate セクション内に明記し、LLM がこのルールに従うよう具体例付きで記述する。複雑な正規表現処理ではなく、LLM が文脈を理解して書き換えることを前提とする。

### Minor observations

- `skills/doc/SKILL.md` は既存で 713 行と大きく、translate セクションを末尾に追記する形で肥大化リスクあり。将来的には `skills/doc/translate-phase.md` への Progressive Disclosure 分離を検討する余地がある（tech.md の Core/Domain 分離方針に準拠）

### Judgment rationale

- **フロントマター除去 vs 保持**: 保持案も検討したが、`Document Traversal common procedure` が `type:` を Grep する設計上、翻訳ファイルが SSoT 候補として混入しステータス表示を壊す。除去して純粋なミラーとする方針で SSoT 衝突を根本回避
- **サフィックス方式（`.ja.md`）vs ディレクトリ分離（`docs/ja/`）**: サフィックス方式は `docs/*.md` Glob でも引っかかり除外ロジックが複雑化する。ディレクトリ分離の方が除外パターンが単純（`docs/ja/` の prefix チェック1回で済む）
- **コミット粒度（1コミット）**: ファイル単位のコミットはレビュー時の差分追跡を難しくするため集約。アトミック性を優先

### Uncertainty resolution

- **LLM 翻訳の品質一貫性**: 初回実装では解決しない。用語集による制御は将来の拡張案として記録。実運用後に人間レビューでフィードバックを収集する方針
- **`doc sync --deep` 除外が効くか**: Implementation Step 4 で除外条件の追加を明記。post-merge で実際に `/doc sync --deep` を実行して吸収候補に上がらないことを確認する（post-merge 条件3）
